# ConnectPad 项目记录

## 目标

提供精简的 Windows 控制台，复用 scrcpy 4.1 完成 Android 11+ 的 USB / Wi-Fi 低延迟投屏与键鼠控制。

## 固定决策

- .NET 8 WPF，Windows x64 便携发布。
- 无 Android APK、无账号、无互联网串流。
- scrcpy 4.1 + ADB；USB 保持 H.264，Wi-Fi 优先硬件 H.265 并自动回退 H.264；60 FPS、最长边 1920、无音频。
- UI 只保留连接所需控件和简短状态。

### 2026-07-18 — 阶段 6：无线低延迟第一阶段实现

- Wi-Fi 启动前调用 scrcpy 4.1 的编码器清单，仅选择标记为硬件的 H.265 编码器；当前实测配置为 H.265 4 Mbps。
- 无硬件 H.265 或检测失败时使用 H.264 8 Mbps；H.265 启动阶段失败时只回退一次，不形成重试循环。
- 无线配置保持最长边 1920、最高 60 FPS、无音频，并通过 `--no-downsize-on-error` 禁止静默降分辨率；USB 参数序列保持原样。
- best-effort 读取 `adb shell cmd wifi status`，解析频段、RSSI、发送速率及发送重试/丢失占比；诊断失败不阻止投屏。
- 增加 `scripts/benchmark-wireless.ps1`，默认覆盖基线和 H.265 6/5/4 Mbps 的三轮五分钟测试，结果写入已忽略的 `artifacts/benchmarks`。
- QUIC Datagram/FEC 第二阶段仍受门禁约束：只有第一阶段在固定 2.4 GHz 环境中未达到 P95 延迟降低 40%，且画质与帧率门槛通过后才进入。

### 2026-07-18 — 阶段 6A：自动化无线基准完成

- 在 2437 MHz 热点上完成 H.264 8 Mbps 与 H.265 6/5/4 Mbps 各三轮、每轮 300 秒的动态滚动测试，12 轮退出码均为 0。
- H.265 6/5/4 Mbps 的平均报告帧数相对基线分别为 +1.8%、+1.3%、+6.2%，总跳帧分别为 27、9、5，H.264 基线为 214。
- 自动结果位于 `artifacts/benchmarks/20260718-205421`；H.265 4 Mbps 固定文字区域相对 H.264 8 Mbps 抽帧的平均 SSIM 为 0.99949、PSNR 为 55.83 dB，未见新增文字模糊，因此应用内采用 4 Mbps。高速相机显示延迟样本和对齐片源 VMAF 尚未采集，QUIC/FEC 门禁仍不打开。
- 版本提升为 `v0.2.0`，重新生成便携 ZIP，并通过包内 `--self-test`；最终 SHA-256 在发布构建后记录。
